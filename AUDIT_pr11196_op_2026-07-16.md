# Op-expert audit of vllm-ascend PR #11196 (`sparse_attn_sharedkv`) + our test harness

Scope: (1) exactly what #11196 changes in the compiled op, (2) is each change correct, (3) are OUR
tests written correctly. Source: `/workspace/dspark_extract/pr11196.diff` + the op tree in
`va-src/csrc/attention/sparse_attn_sharedkv/`. Every claim → `file:line`.

## Verdict

- **#11196's op changes are correct, complete, and consistent.** It does one thing to the kernel:
  **enable a non-causal right window (`win_right > 0`)** in the four places that had it hardcoded to
  causal. The math checks out (I verified the mask formula against the PR's own window test, and the
  essential `oriMaskRight` clamp).
- **#11196 does NOT touch the softmax / sink / matmul.** Those I read directly: the **sink is correctly
  implemented** (flash-softmax initialized with `max=sink, sum=1.0` — the exact "virtual sink key")
  and **internal compute is fp32**. So neither the window nor the sink nor precision explains our
  measured ~2e-2 `PROD vs REF`.
- **Our tests are correctly written** — our reference IS the PR's own reference (byte-identical), our
  window matches the op's formula, our scenario is faithful, and `OURS vs REF = 1.5e-4 (bf16) /
  1.9e-5 (fp16)` corroborates both. So the ~2e-2 is a **real op deviation** our harness correctly flags.
- **The PR's OWN unit tests never run the compiled op vs the reference** — a real coverage gap; that is
  why the ~2e-2 was never caught upstream and our harness is the first to see it.

## 1. Complete inventory of #11196's op changes (4 source files, all tiny)

| file | change | what it does |
|---|---|---|
| `op_host/sparse_attn_sharedkv_tiling.cpp:1365,1368` | `oriWinLeft_ != 127 → < 0`; `oriWinRight_ != 0 → < 0` | relax the host-side gate so `win_right ≥ 0` is accepted (was causal-only) |
| `op_kernel/arch32/sparse_attn_sharedkv_swa_kernel.h:692` (+ loop-cond `:387`, split `:711`) | add `oriMaskRight = Min(oriMaskRight, actOriS2Size-1)`; `hasOriWindow = oriMaskRight >= oriMaskLeft`; empty-window `oriSplitNum` guard | SWA path: honor `win_right` and **clamp the right edge to the KV length** |
| `op_kernel/arch32/sparse_attn_sharedkv_scfa_kernel.h:718` (+ `:379`, `:726`) | IDENTICAL clamp + guards | compressed path: same fix, kept consistent with SWA |
| `metadata_aicpu/..._metadata_aicpu.cpp:266` | `nextToken_ = 0 → (winRight_ > -1 ? winRight_ : INT64_MAX)`; relax gate + add `winRight_<0` check | AICPU sparse-mask builder: encode the right window (`preToken_=winLeft_`, `nextToken_=winRight_`) |

The pre-existing line `oriMaskRight = actOriS2Size − actS1Size + s1EndIdx + oriWinRight` already added
`oriWinRight`; upstream just never allowed it to be non-zero (the gate). #11196's real content is the
**gate relaxation + the `Min(...)` clamp** (without the clamp, a non-zero `win_right` overshoots the KV
buffer). Everything else in the PR is Python (new `ops/dspark_attention.py`, `deepseek_v4_dspark.py`,
`dspark_proposer.py`, `attention/dsa_window.py`) — not the compiled op.

## 2. Is each op change correct? Yes.

- **Window math** — verified against the PR's own test `tests/ut/spec_decode/test_dspark_attention_semantics.py`:
  `left = max(s2−s1+q−winL, 0)`, `right = min(s2−s1+q+winR, s2−1)` (its `_window_bounds`), which is
  exactly the kernel's `oriMaskLeft/oriMaskRight` after the clamp. Its
  `test_dspark_sas_window_covers_context_and_full_draft_block` asserts this window covers `[context |
  full draft block]` for every query — i.e. at the DSV4 geometry (block=5, KV=133) the window is a
  **no-op** (all 133 keys visible), which is what makes op == dense reference. Confirmed independently
  in our `diag_sas_window.py`.
- **The clamp is essential and right.** win_right=4, KV=133 → right edge = `133−5+i+4 = 132+i` → without
  `Min(·, 132)` it reads keys past the buffer. With it, right=132 (all keys). Correct.
- **Metadata `nextToken_ = winRight_`** correctly injects the right window into the sparse-mask builder
  (FlashAttention `preToken/nextToken` convention). Correct and consistent with the kernel.

## 3. The sink (my ~2e-2 suspect) — read directly, it is CORRECT

`op_kernel/arch32/sparse_attn_sharedkv_swa_block_vector.h`:
- `using T = float` (`:33`, "高精度模式") ⇒ `SINKS_T = T = float` (`:37`) — **the sink is fp32**, not the
  input dtype. (My earlier "sink in bf16" worry was wrong.)
- `InitSoftmaxDefaultBuffer` (`:314`): `CopySinksIn()`, then `softmaxSumDefaultUb = R0 = 1.0` (`:318`,
  `R0` at `:119`).
- `SliceAndContactSinksValue` (`:295`): `WholeReduceMax(softmaxMaxDefaultUb, sinksBrcbUb, …)` ⇒ the
  flash-softmax **initial max = sink**, **initial sum = 1.0 = exp(sink−sink)**.
- The online `SoftmaxFlashV2` (`:352`) then folds real keys in, rescaling the sink term correctly:
  final `sum = exp(sink−max) + Σ exp(scores−max)`; the sink contributes to the denominator only (no V).
  This is **algebraically identical** to `_dspark_attention_reference`'s
  `probs = e / (e.sum() + exp(sink−max))`. Correct.

So window ✓, sink ✓, fp32 ✓. #11196 touched none of these except the window (which is a correct no-op
here). The ~2e-2 is therefore **not** any of the suspects I chased earlier (window, staleness, sink,
precision) — see §5.

## 4. Are OUR tests correct? Yes.

- **Reference identity.** Our `fused_sas_vs_reference_parity.py` / `ours_vs_production.py` compare
  against `vllm_ascend.ops.dspark_attention._dspark_attention_reference` — the op's OWN reference, and
  byte-identical to the sink formula in the PR's semantic test (`:46-49`). So "REF" is the op's spec.
- **Window identity.** `diag_sas_window.py::masked_reference` masks on absolute positions
  `kp∈[qp−winL, qp+winR]`; in our scenario (context `[0,W-1]`, block at `[W, W+B-1]`, base=W) the
  absolute position equals the packed index, so this is exactly the kernel's right-aligned
  `[s2−s1+q−winL, s2−s1+q+winR]`. Equivalent — verified.
- **Scenario faithful.** shared-KV/MLA (num_kv_heads=1, one latent broadcast — after the per-head-K bug
  fix), `win_right = block−1` at the DSV4 inference geometry (block=5 ⇒ 132/4), per-head fp32 sink.
- **Cross-check.** `OURS vs REF = 1.5e-4 (bf16) / 1.9e-5 (fp16)` — our independent Triton kernel and the
  op's reference agree, corroborating both. So when the harness reports `PROD vs REF = 1.60e-2 / maxAbs
  1.05`, that is a genuine op-vs-spec deviation, not a harness artifact.

**One caveat on coverage:** our harness drives the op through impl A's per-block `_call_dspark_sas_block`
(`layout="TND"`, `num_kv_heads=1`, KV=133). The real serve's `dsa_v1.py` path uses `layout="PA_ND"`
(paged) — the SAME op binary, fed differently. Our test is a valid op invocation, but a residual could
in principle be specific to the TND/per-block invocation and absent in the paged path. Only an
end-to-end run distinguishes.

## 5. The ~2e-2, honestly

Ruled out by measurement and by reading the source: **the window** (PROD closer to non-causal than
causal; clamp is correct; window is a no-op here), **precision** (fp16==bf16 error; our bf16 kernel hits
1.5e-4), **a stale/partial build** (recompiled `.o` fresh at 17:49, PROD byte-identical; per-position
error uniform, not the overshoot signature), and **the sink** (correct flash-init, §3). It is uniform
across block positions and precision-independent — a narrow, systematic residual in the pre-existing
softmax/matmul path that I cannot pin to a source line, and that #11196 did not introduce.

**Recommended arbiter (not more source-diving): end-to-end `accept_len`.** Run the RELEASED DSV4 draft
on the serve → gsm8k accept_len at num_spec=5 (bar 3.94). Spec-decode verifies every draft token, so a
~2e-2 draft-attention deviation typically only lowers acceptance, not correctness. If `accept_len ≈
3.94`, the residual is tolerated end-to-end and is not worth chasing further; if it's low, the op is
biting and the paged-path invocation (not this harness) is where to look next.

## 6. UPDATE — the ~2e-2 is a TND-wrapper bug; PA_ND (paged) is the real serve path

Root-caused on real A3 (2026-07-16). A KV-length sweep (`ours_vs_production.py`, vary `WIN`, `KV=WIN+BS`):

| KV | `PROD vs REF` meanAbs |
|---|---|
| **128** | **1.55e-4  (allclose=True)** ✅ |
| 133 (DSV4 real) | 1.60e-2 |
| 144 | 3.59e-2 |
| 256 | **NaN** |

Correct only at KV≤128 (one KV tile, `s2BaseSize`); error grows with the second-tile size and NaNs at
256 ⇒ the op's **multi-KV-tile flash-softmax accumulation is broken** — BUT only in the way OUR harness
calls it. The harness uses impl A `_call_dspark_sas_block` with **`layout_kv="TND"`, `seqused_kv=None`,
no block_table** (`dspark_attention.py:168,178`). The op's DESIGNED interface is **PA_ND (paged)**:
README says `layout_kv` PA_ND, `ori_kv=[block_num, block_size(=16·k), 1, D]`; the real serve
`dsa_v1.py` calls it with `layout_kv="PA_ND"`, `seqused_kv=self.seq_lens` (=133), `block_table`
(`dsa_v1.py:753,762,713`). Sink was RULED OUT first: `NOSINK=1` left `PROD vs REF` unchanged at 1.6e-2,
and per-head error is uncorrelated with sink.

**Resolution of "then it couldn't measure accept length":** the released-draft AL 3.94 comes from the
**PA_ND fused path**; our ~2e-2 is the **TND eager wrapper** (impl A), a different (and evidently
under-tested for KV>tile) calling convention. No contradiction. ⚠ The default env
(`VLLM_ASCEND_DSPARK_USE_STANDARD_DSA=0`) routes the draft model's attention through that SAME eager
TND path (`deepseek_v4_dspark.py:361` else-branch → `_run_dspark_attention` → `dspark_attention`), so a
default-config serve would hit this at KV=133 → flag to the serve team to run PA_ND (`=1`) or fix the
TND path. NEXT: a PA_ND paged harness to confirm the op is correct at KV=133 in the real path.

## TL;DR for the trainer chain

#11196's op changes = a correct, minimal, consistent enablement of the non-causal `win_right`. The op's
window+sink+dtype are correct by inspection. Our tests are correct and are the ONLY thing that actually
exercises the compiled op vs its reference (the PR's own UT does not). The residual ~2e-2 is real but
narrow, is NOT window/sink/precision/staleness, and should be judged by end-to-end `accept_len`, not by
more isolated parity.
